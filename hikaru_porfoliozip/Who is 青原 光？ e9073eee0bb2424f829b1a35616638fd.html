OneHalf

アプリの説明

二分の一の確率で正解するボタンを連続で何回選び続けられるかをランキングで競い合うゲームです。

なぜ作った？

Swiftを学んではいたものの、これまで一度もアプリを実際に作ってきたことがなかったため、実際にアプリを作成することでこれまで学んだことをアウトプットしようと考え作成しました。

また作成したアプリをリリースすることによるリリース経験を積むことも目的としました。

学んだこと

1. アプリを開発する経験

自分で作成した画像を使う方法や、GameKitの使い方、アプリのリリース方法などを学ぶことができました。

2. アーキテクチャの必要性を実感

また、アーキテクチャを意識せずにコード書いてしまったせいで、コードに小さな変更を加えるだけでも多くの箇所を修正しなけらばならなくなるという苦い経験をしました。

このことから、アーキテクチャの必要性を知ることができました。

使用技術

言語：Swift

フレームワーク・ライブラリ：UIKit, GameKit

その他：Git, GitHub

GameKit

単なるゲームアプリでは単純すぎたため、ユーザー同士でスコアを競うことができるランキングを実装しようと考えました。

ランキング機能については、サーバーを立てたりFirabaseを活用してバックエンドを実装することも考えましたが、それらの方法は学習コストや保守のためのコストが高いというデメリットがあり、導入するには至りませんでした。

GameKitのランキング機能は実装したい機能の要件を満たしており、また学習コストや保守のコストも低いと判断したため、GameKitを使用することにしました。

Tune the Tone

アプリの説明

RGBスライダーを動かして、画面に表示される色と同じ色を作るゲームです。

作った色のRGB値と正解のRGB値とのマッチ度を全世界のユーザと週間ランキングで競うことができます。

なぜ作った？

勉強したSwiftUIを使って何かアプリを作ってみたいと思い作りました。

また、OneHalfの開発経験から学んだことを活かして、初めてアーキテクチャ（MVVM）を意識して開発したアプリでもあります。

学んだこと

1. UIのモジュール化によるメリット

UIをモジュール化し、再利用することによる利便性を学びました。

例えば、ゲーム中の画面に表示される下の画像のようなスライドバーは、赤・緑・青のそれぞれをViewに直接記述するのではなく、ColorSlider.swiftという別ファイルにモジュールとして取り出すことでコードの可読性が向上し、再利用性が高まりました。

この経験から、コードのモジュール化の重要性を理解することができました。

2. MVVM

Model、View、ViewModelを分離することにより、コードの可読性が上がることを体験することができました。

また、このアプリは個人で開発したのでメリットを享受することはできませんでしたが、チームで開発をする際にはコードが分離していることによって一つのアプリを並行して開発しやすくなりそうだなと感じました。

使用技術

言語：Swift

フレームワーク・ライブラリ：SwiftUI, GameKit

その他：MVVM, Git, GitHub

MVVM

SwiftUIのプロパティラッパーによるデータバインディングが、MVVMのViewModelによるViewの自動更新と相性がいいと考えMVVMを選択しました。

WristGraph

アプリの説明

ユーザ名を登録したGitHubアカウントのコントリビューショングラフを確認できるアプリです。

iPhoneアプリ、Apple Watchアプリ、Apple Watchコンプリケーションに対応しています。

まだ完成はしていないためApp Storeへは公開しておりません。

なぜ作った？

いつでもどこでも、Apple Watchからコントリビューショングラフを確認したいと思い、アプリを探したのですが、当時どのアプリもApple Watchやコンプリケーションに対応していなかったため、自分で作ることにしました。

iPhoneアプリとApple Watchアプリとの間でのデータのやり取りの仕方や、Widgetの作り方のノウハウを知ることができました。

学んだこと

1. iPhone、Apple Watch間でのデータのやり取り

Watch Connectivityというフレームワークを使用して、iPhoneからApple Watchへデータを送信する方法を学ぶことができました。

2. APIを使用したデータ取得方法

APIを使用してGitHubのサイトからコントリビューショングラフのデータを取得する方法を学ぶことができました。

また、本来実装する必要のあるURLSessionなどによるAPI呼び出し処理を、簡単に記述することができるようになるMoyaというライブラリの使い方も学ぶことができました。

使用技術

言語：Swift

フレームワーク・ライブラリ：SwiftUI, WatchKit, Watch Connectivity, WidgetKit, Moya

その他：MVVM, Git, GitHub

Moya

Swiftでは、APIなどによる通信機能を実装する際には通常URLSessionを用いた複雑な処理を実装する必要があるが、Moyaを使用すればこれらの処理を簡単に記述することができるため、実装コストや保守コストを下げることができるため導入した。

PredicTalk

アプリの説明

ユーザーの発話を認識し、言語モデルによって続きの文章を予測し画面、もしくは接続したARグラス上に表示するアプリです。

英会話中など、ネイティブレベルでない言語を使って会話している時にこのアプリを使うことで、「この後どんな単語・文法を話せばいいんだろう」というように会話に詰まってしまっても、画面上、もしくは視界上に次に話すべき内容が表示されるので、自分から会話を再開させることができるようになります。

なぜ作った？

従来の翻訳アプリには翻訳した文章を相手に見せたり、合成音声でアプリに話させて使用するため、ユーザー自身が英語を話す機会を奪ってしまっています。

そのため、相手の目を見て、自分の口から英語を話すことによる楽しさを味わうことができません。

「スマホではなく相手の目を見て、自分の口から英語を話したい！けど、英語を完璧に話し切る自信はない！」

このようなニーズに応えるために、本アプリを開発しました。

学んだこと

1. Speechフレームワークの使い方

音声認識を実装するため使ったSpeechフレームワークについて学ぶことができました。

リアルタイムで音声認識をするためのストリーム処理を実装するのに苦労しましたが、公式ドキュメントや他の開発者による実装例を参考にして実現することができました。

2. LicenseListの使い方

使用したライブラリのライセンス情報を自動的に取得し、表示してくれるLicenseListというライブラリの使い方を学ぶことができました。

使用技術

言語：Swift

フレームワーク・ライブラリ：SwiftUI, Speech, Moya, LicenseList

その他：Git, GitHub

言語モデル

本アプリではユーザーの発話の続きを予測するために言語モデルを使用しています。

開発当初、API経由でGPT-4を利用することを考えましたが、ネットワークによる遅延でリアルタイム性が損なわれるのではないかと考え、軽量な言語モデルをデバイスに組み込むことにしました。

そこで、MobileBERTといったモバイル端末用に軽量化された言語モデルを、iPhone上で動作するようにCoreMLという形式に変換しアプリに組み込みました。

しかし、実際に使用してみると軽量なモデルなため予測結果の精度が低く、またiPhoneのスペック不足によって推論に時間がかかってしまうことが判明しました。

APIを使用する際のネットワーク遅延よりも推論による遅延の方が大きかったため、結局API経由でGPT-4を使用することにしました。
